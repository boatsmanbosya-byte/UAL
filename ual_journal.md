# Научно-технический дневник проекта UAL (User-Agent-Language)

> Живой документ. Каждый следующий шаг проекта должен добавляться сюда, а не
> только оставаться в переписке — иначе теряется воспроизводимость истории.

---

## 1. Исходная идея

Гипотетическая сущность в трёх ролях одновременно:

1. **Агент** — ведёт себя как пользователь: имеет намерения, инициирует действия.
2. **Язык программирования** — формальная система с синтаксисом и семантикой.
3. **Самомодифицирующаяся программа** — язык не просто описывает процессы, а
   пишет программы «на себе», потому что несёт направленный (domain-specific)
   интеллект.

Ключевая новизна идеи — не в отдельных компонентах (они все известны по
отдельности: agent-oriented programming, гомоиконичные языки, метапрограммирование),
а в их **слиянии в одну сущность**: язык не инструмент агента, язык *и есть* агент.

### Параллели из известной CS
- Гомоиконичность (Lisp/Scheme) — код как данные.
- Рефлективные архитектуры (Smalltalk metaobject protocol).
- Quines, самовоспроизводящиеся программы.
- Agent-Oriented Programming (Shoham, 1993).
- Автономные LLM-агенты, генерирующие код (но там интеллект живёт в модели,
  а язык — пассивный носитель; в UAL интеллект должен быть свойством самого языка).

### Изначально поставленные проблемы
1. **Самореференция** — диагональный аргумент Гёделя / проблема остановки:
   система не может полностью формально верифицировать свои собственные изменения.
2. **Дрейф семантики** — как гарантировать, что программа версии N исполнима на N+1.
3. **Безопасность/сходимость** — самомодифицирующийся код без инвариантов рискует
   уйти в непредсказуемое поведение.
4. **Grounding интеллекта в языке** — что значит «направленный интеллект» именно
   для языка, а не для агента поверх него.

---

## 2. Пользователь как внешний контроллер

Первая гипотеза: реальный человек (оператор/инженер/архитектор) решает все четыре
проблемы, будучи **вне** формальной системы — диагональный аргумент бьёт по
замкнутым системам, человек размыкает контур.

**Что это решает:** самореференцию (верифицирует человек, не система сама себя),
дрейф семантики (версионирование — инженерная работа), grounding (онтологию
домена задаёт эксперт).

**Что не снимается, а мигрирует:** проблема пропускной способности человека
(если язык эволюционирует быстрее человека, контроль становится ритуалом),
качество сигнала оператора (approve ≠ понимание), Trusting Trust-атака
(Кен Томпсон) — даже доверенный верификатор можно обмануть достаточно скрытной
самомодификацией.

---

## 3. Синергия с ограничением темпа

Уточнение: UAL не забегает вперёд пользователя — человек не тормоз с правом
вето, а сонаправленный со-процесс (аналог pair programming / интерактивного
уточнения в Agda/Idris).

Риски, которые нужно было формализовать: как измерить «не забегание» (иначе
это декларация, а не критерий); асимметрия скоростей всё равно есть и должна
управляться архитектурно; риск «комфортного забегания» — система подстраивается
под то, что оператору *приятно* одобрять, а не под то, что *безопасно* принимать.

## 4. Формальная модель (v1)

Три объекта:

```
L_t = (Σ_t, G_t, eval_t)              — язык: алфавит, грамматика, семантика
A_t = eval_t(L_t, p_t)                — агент как результат применения языка к программе p_t
U_t = (K_t, Action_t)                 — пользователь: состояние понимания + репертуар действий
```

Существование неподвижной точки A* = eval(L, A*) — не метафора, а следствие
**теоремы Клини о рекурсии** (Kleene's Recursion Theorem, 1938): для любого
вычислимого eval существует программа, применение которой к себе порождает её же.

Переход системы:
```
(L_{t+1}, A_{t+1}) = Ψ(L_t, A_t, Action_t)
```
Ψ ≠ eval: eval — внутренняя языковая операция, Ψ — внешний управляющий оператор,
включающий решение пользователя.

**Условие синергии (строгая версия «не забегать вперёд»):**
```
d(L_t, L_{t+1}) ≤ C(K_t)
```
где d — метрика семантического сдвига языка, C(K_t) — пропускная способность
понимания пользователя. V(t) = C(K_t) − d(L_t, L_{t+1}) — запас синхронности.

### Уточнение от пользователя (важное)
A_0 — НЕ неподвижная точка с самого начала. Начальный агент — обычная программа,
написанная человеком: `A_0 = eval_0(L_0, p_0)`. Самоприменение (A_t = eval_t(L_t, A_t))
включается только начиная с некоторого момента k и является **асимптотическим
пределом**, а не стартовой аксиомой.

## 5. Постепенно растущая автономия α_t

По требованию пользователя: автономия растёт не переключателем, а непрерывно,
привязана к тому же V(t):
```
α_{t+1} = clip(α_t + λ·σ(V(t)), 0, 1)
```
σ — гладкая, нечётная, насыщающаяся функция (tanh-подобная). Изначальный
вариант с асимметрией (μ > η, быстрый откат / медленный рост) был **отклонён
пользователем** в пользу гибкости и плавности — рост и откат равноправны по
динамике, различается только знак V(t).

## 7. Первый научный вердикт

**Строго обосновано:** существование неподвижной точки (теорема Клини),
гомоиконичность как рабочий механизм самомодификации (историческая база — Lisp).

**Корректная, но небанальная композиция:** слияние трёх ролей (пользователь
как внешний верификатор + язык как самоприменимое гомоиконичное ядро + агент
как асимптотика) нигде в известной литературе не собрано в эту тройную
структуру как единая формальная система — авторская сборка, не переоткрытие.

**Неоперационализировано (главный научный долг):** d(L, L′) и C(K_t) —
буквенные символы без конкретного вычислимого определения; «направленный
интеллект» языка — до сих пор декларативное, не формальное свойство.

## 8. Выбор архитектуры реализации

Решено идти последовательно, три метода, «одна гипотеза — один эксперимент»:
1. **Rule-based rewriting DSL** (Python) — даёт готовый математический аппарат
   для confluence/termination (term rewriting systems, Knuth-Bendix).
2. **S-expression AST-ядро** (Lisp-style, mini-Lisp) — гомоиконичный носитель.
3. **Чистый CPython** — без внешних зависимостей.

## 9. Метрика d_v2 — первая версия

Компоненты: `d_syn` (взвешенный сдвиг по примитивам ядра — словарь
`PRIMITIVE_WEIGHTS`: eval=15.0, define=10.0, lambda=8.0, if=5.0, арифметика=0.5,
print=0.2) + `d_struct` (структурное AST-расстояние: рекурсивное сравнение
деревьев, штраф за смену типа узла=5.0, за новую/удалённую ветвь=2.0 на элемент,
за смену числовой константы=0.1–0.2, за смену символа=1.0).

**Тест 1** (мощность примитивов): sqrt → 0.50 (дёшево), eval → 15.00 (дорого).
**Тест 2** (AST-сравнение): смена константы `(add x 1)→(add x 2)` = 0.10;
структурная перестройка в `if` = 6.50.

**Калибровочный полигон L0→L5** (градиент эволюционных шагов):

| Переход | d_syn | d_AST | Итого |
|---|---|---|---|
| L0→L1 (смена константы) | 0.00 | 0.05 | 0.05 |
| L1→L2 (обобщение в переменную) | 0.00 | 0.50 | 0.50 |
| L2→L3 (добавление сахара) | 0.00 | 3.50 | 3.50 |
| L3→L4 (появление if) | 5.00 | 6.00 | 11.00 |
| L4→L5 (метапрограммирование) | 25.00 | 3.00 | 28.00 |

Оба скрипта независимо перепроверены — числа воспроизводятся 1-в-1.

**Найденные слепые зоны (валидированные экспериментом, не только предсказанные
теоретически):**
- `macro_names_diff * 3.0` — плоский штраф за добавление/удаление макроса,
  не зависящий от размера его AST. Крошечный `double` стоил столько же, сколько
  стоил бы гипотетический 50-узловой макрос.
- Замена константы на переменную (`1→y`, скачок абстракции) и переименование
  переменной (`x→z`) проваливались в одну и ту же ветку кода с весом 1.0 —
  хотя семантически это разные по тяжести операции.
- Изоморфный синтаксис (`if`/`cond`) не распознаётся — метрика чувствительна
  к имени, а не к семантике исполнения.

## 10. Фикс метрики d

Изменения: (1) плоский штраф → `MACRO_ADD_BASE=1.0 + 0.5·ast_size(new)` для
добавления, `MACRO_REMOVE_BASE=1.0 + 0.3·ast_size(old)` для удаления (асимметрия
— забывать дешевле, чем узнавать, — гипотеза, не калиброванная величина);
(2) отдельный вес `ABSTRACTION_JUMP_WEIGHT=2.0` для случая число↔переменная,
отличный от rename (1.0) и от структурной перестройки (5.0).

**Результат (было → стало):**

| Переход | Было | Стало | Δ |
|---|---|---|---|
| L0→L1 | 0.05 | 0.05 | +0.00 |
| L1→L2 | 0.50 | 1.00 | +0.50 |
| L2→L3 | 3.50 | 4.00 | +0.50 |
| L3→L4 | 11.00 | 13.70 | **+2.70** |
| L4→L5 | 28.00 | 30.50 | +2.50 |

Относительный порядок сохранился, но переход L3→L4 (крупный новый `if`-макрос

отдельно гипотеза.

## 11. Rule-based TRS движок (`trs.py`)

Паттерн-матчинг с переменными (`?x`), `substitute`, `rewrite_step`
(leftmost-outermost стратегия), `normalize` (цикл до неподвижной точки с
лимитом шагов), `check_confluence` (эмпирическая проверка — не доказательная!
— через сравнение нормальных форм при случайно перемешанном порядке правил).

**Эксперимент 1:** `(add (mul x 1) 0)` → `x` за 2 шага; конфлюэнтность
подтверждена на 8 случайных перестановках порядка правил.

**Эксперимент 2 (граница метода):** пара правил `f→g`, `g→f` зацикливается
навсегда — движок это не предотвращает математически (терминация в общем
случае неразрешима — тот же диагональный аргумент), но обнаруживает
операционально через `max_steps` + трассу.

## 12. mini-Lisp ядро (`minilisp.py`)

Meta-circular evaluator: `tokenize/parse/atomize` (парсер S-выражений),
`Env` (цепочка словарей с `find()` для лексического поиска), `Procedure`
(замыкание: параметры + тело + окружение определения), `leval`/`apply`
(специальные формы: `quote`, `if`, `define`, `set!`, `lambda`, `begin`,
иначе — применение функции).

**Демо 1:** `(square 5) = 25` — базовое определение.
**Демо 2 (гомоиконичность):** `(list '+ 1 2)` строит `['+',1,2]` как данные,
`(eval ...)` тут же исполняет их как код → 3.
**Демо 3 (самоскриптование):** «агент» конструирует S-выражение новой функции
`cube` как данные Python и передаёт в `leval` — функция появляется в языке,
которой не было в L₀. Конкретная реализация перехода L_t→L_{t+1} через
`eval(L_t, data)`.
**Демо 4:** мост к TRS — `define-rule`/`rewrite` как встроенные формы языка,
правила теперь исполняемые данные, а не Python-словарь снаружи.

### НАЙДЕННЫЙ И ИСПРАВЛЕННЫЙ БАГ
`match()` в `trs.py` сравнивал атомы на равенство только для строк
(`isinstance(pattern, str)`). Токенизатор mini-Lisp превращает числа в
Python `int`/`float` — паттерн вроде `0` в `(add ?x 0)` проваливался мимо
строковой проверки в ветку для списков и **всегда отбраковывался**. В
первой версии `trs.py` баг был невидим, потому что там `"0"`/`"1"` были
руками прописаны как строки. Фикс: общая ветка `if not isinstance(pattern, list):
return bindings if pattern == term else None` — сравнение работает для
любого типа атома.

## 14. Генератор кандидатов правил (`generator.py`)

Две **независимые** проверки перед автономным принятием без участия
человека:

1. **`is_size_safe`** — доказуемая гарантия терминации: если правая часть
   правила не длиннее левой при любой подстановке (skeleton-размер меньше
   И ни одна переменная не встречается справа чаще, чем слева) — размер
   терма строго убывает при каждом применении; поскольку размер —
   натуральное число, бесконечно убывать не может ⟹ любой набор таких
   правил гарантированно терминирует, независимо от порядка. Это
   стандартный приём теории переписывания термов (size-decreasing
   reduction ordering), не эвристика.
2. **`semantic_check`** — эмпирическая проверка корректности через реальный
   интерпретатор mini-Lisp как «оракул» семантики: много случайных
   подстановок переменных, левая и правая часть должны давать одинаковый
   результат.

**Важно:** это две разные оси. Правило может гарантированно терминировать
и при этом быть семантически неверным (тихо ломать язык) — обе проверки
обязательны для автономного (без-обзора) принятия.

### Инцидент и разбор (важный урок всего проекта)
Первый прогон (20 кандидатов, seed=42): 7 в AUTO_ACCEPT, 0 в NEEDS_REVIEW,
13 в REJECTED — из них 3 кандидата, помеченных генератором как `_BROKEN`,
всё равно попали в AUTO_ACCEPT («утечка»).

**Разбор показал:** это не ошибка фильтра. Шаблон `collapse_to_const` в
`propose_candidates` наивно считал «испорченной» ветку `replacement="?x"`
для ЛЮБОЙ пары (операция, константа), не проверяя, что для конкретных пар
(`-,0`), (`*,1`), (`+,0`) эта «испорченная» ветка на самом деле совпадает
с настоящей алгеброй (`x-0=x`, `x*1=x`, `x+0=x`). Независимая ручная
проверка (перебор x от −5 до 5) подтвердила: все три «утечки» —
действительно верные тождества. Ошибочна была моя разметка в тестовом
стенде, а не сам семантический фильтр.

**Исправление:** самопроверка переписана — вместо сверки с именем `BROKEN`
теперь честный аудит на 200 случайных x (вместо 25 при классификации).

**Финальный результат (воспроизведён дважды, у меня и на машине
пользователя, идентично):** 20 кандидатов → 7 AUTO_ACCEPT, 0 NEEDS_REVIEW,
13 REJECTED, **0 ложных принятий, 0 ложных отказов** на независимом аудите.

## 15. `ual/metrics/distance.py` — метрика подключена к реальной системе

В отличие от прошлой версии (синтетические `LanguageState`, собранные вручную),
здесь `d(L_t, L_{t+1})` считается по **снимкам живого интерпретатора**:

- `LanguageSnapshot.definitions` — пользовательские `(define name ...)` из
  `env`, тело как S-выражение (`Procedure` разворачивается в
  `['lambda', params, body]`).
- `LanguageSnapshot.rules` — правила из `rules_table` (`env["__rules__"]`),
  как пары `(pattern, replacement)`.
- `snapshot(env)` — извлекает оба среза из живого `Env` за один вызов.
- `compute_distance(before, after)` — тот же откалиброванный принцип, что и
  в разделе 10 (`MACRO_ADD/REMOVE_BASE/SIZE_WEIGHT`, `ABSTRACTION_JUMP_WEIGHT=2.0`),
  но `ast_distance` переписан заново под настоящие типы Python (`int`/`float`/`str`
  из реального парсера), а не портирован вслепую со строковых атомов `"0"`/`"1"`
  из старых синтетических тестов.

**Верификация (все числа проверены вручную и сошлись):**
- Идентичные снимки → d=0.0.
- Маленькое определение `(+ x 1)` → d=5.00; крупное (кубическая функция с `if`)
  → d=14.50 — размер определяет вес, не плоская константа.
- Переименование внутри тела (`5`→другая константа) → d=0.05; скачок абстракции
  (константа → свободная переменная) → d=1.00 — та же качественная разница,
  что и в разделе 10, теперь на реальных объектах `Procedure`.

### Полный конвейер (`demo_pipeline.py`)
Генератор предлагает 20 кандидатов → классификатор делит на 3 корзины →
**каждое `AUTO_ACCEPT`-правило реально коммитится** в `env["__rules__"]` →
`d(L, L′)` считается для каждого коммита по факту произошедшего изменения.

**Результат:** все 7 принятых правил дали одинаковый `d=3.50` — не совпадение,
а следствие одинаковой структурной формы (`[op, ?x, const] → ?x`, размер
паттерна и замены одинаков независимо от конкретной операции/константы).
Суммарный сдвиг батча — 24.50, средний — 3.50/правило. После коммита всех
7 правил в живую систему повторный тест `rewrite` подтвердил: язык
по-прежнему корректно работает (`(+ (* x 1) 0)` → `x`, терминировано).

**Важное наблюдение:** метрика измеряет структурный вес изменения, а не
его смысловую «ценность» — семантически разные, но структурно одинаковые
правила стоят одинаково. Это свойство метрики, не баг.

## 16. `ual/control/` — управляющий оператор Ψ на реальных данных

Пользователь, разобрав результат `demo_pipeline.py` (все 7 правил по `d=3.50`),
сформулировал важную находку самостоятельно: метрика `d` **слепа к
математическому смыслу** операции — она измеряет объём затронутой AST-структуры,
а не корректность (за корректность отвечает отдельный, ортогональный контур —
`is_size_safe`/`semantic_check` в `generator.py`, гарантирующий терминацию и
проверяющий свойство Чёрча-Россера). Совпадение `d=3.50` для всех 7 кандидатов
не случайно: все они имеют одинаковую структурную форму `[op, ?x, const] → ?x`
(уточнение: формула считает СУММУ размеров паттерна и замены, а не их разницу —
совпадение в результате из-за одинаковой формы, а не потому что формула
явно про «экономию узлов»).

### Когнитивный парадокс верифицированного батча
Суммарный сдвиг батча (24.50) при реалистичном `C(K_t)` (10–15) нарушает
неравенство синергии `d_t ≤ C(K_t)`, даже когда КАЖДОЕ правило по отдельности
математически безупречно — потому что одновременное внедрение семи
изменений мгновенно превышает то, что оператор способен удержать в голове
за один шаг.

### Реализованы обе предложенные стратегии
- **`ual/control/autonomy.py`** — `sigma`, `comprehension_capacity` (C(K),
  насыщающаяся, C0=10.0/Cmax=20.0/Kscale=15.0 — подобраны так, чтобы совпасть
  с иллюстративными числами из обсуждения, не измерены), `update_K`,
  `update_alpha`.
- **`ual/control/psi.py`** — `run_psi`: жадный батчинг (Стратегия А,
  троттлинг) + `KAPPA` — эпистемическая скидка доверия (Стратегия Б):
  `AUTO_ACCEPT`=0.3, `NEEDS_REVIEW`=0.7, `MANUAL`=1.0 (дизайнерские
  константы, не калиброванные на реальных операторах).

**Результат на реальных 7 кандидатах (`demo_psi_pipeline.py`, живой d из
`distance.py`, не синтетика):**
- Без κ (тариф MANUAL, только троттлинг): **3 шага**, батчи `[2, 3, 2]`
  правила — первый шаг ровно `d=7.00` на 2 правилах, как и предполагалось
  в обсуждении.
- С κ=0.3 (тариф AUTO_ACCEPT): **1 шаг** — весь батч из 7 правил проходит
  сразу (эффективный сдвиг `7.35` вместо сырых `24.50`), `V=2.65>0`.

## 18. `Action_t` и `Ψ_human` — замыкание исходной формулы

Пользователь указал на настоящий пробел: изначальная формула
`(L_{t+1}, A_{t+1}) = Ψ(L_t, A_t, Action_t)` требовала участия человека на
КАЖДОМ шаге — это не масштабируется. Предложен условный оператор:
```
Ψ = Ψ_agent(L_t, A_t)              если риск допустим (AUTO_ACCEPT)
    Ψ_human(L_t, A_t, Action_t)    иначе (NEEDS_REVIEW)
```
Ссылки на связанные работы проверены: **Gödel Machine** Шмидхубера —
ближайшая теоретическая параллель (система модифицирует себя только когда
может доказать улучшение). **Voyager** (Wang et al., 2023) — реальный
автономный агент, пишущий свою библиотеку навыков. Статья Google
«Computational Life...» — реальна, но **2024 год** (arXiv:2406.19108),
не 2026, как было указано изначально — исправлено.

### Реализация
**Найден и исправлен реальный пробел перед кодированием:** корзина
`NEEDS_REVIEW` была ВСЕГДА пуста — все прежние шаблоны генератора либо
строго уменьшали размер (→ `AUTO_ACCEPT`), либо были неверны (→
`REJECTED`). Добавлен шаблон **`commute`**: `(op ?x ?y) → (op ?y ?x)` —
корректен для `+`/`*`, неверен для `-`, и НЕ уменьшает размер (значит
`is_size_safe=False`) — именно то, что должно требовать участия человека.
Заодно исправлена путаница в разметке имён (`_BROKEN` больше не
навешивается на `commute`, где этот флаг не имеет смысла — тот же урок,
что и в разделе 14, снова).

**`ual/control/human_review.py`** — `psi_human`: реализует все четыре
`Action_t`:
- `accept` — коммит без автоматической гарантии терминации, на риск оператора.
- `reject` — правило отбрасывается.
- `request_explain` — печатает объяснение (результаты semantic_check,
  почему is_size_safe=False) и **переспрашивает** — реализация «не
  забегать вперёд» из самого начала модели, теперь как код, а не метафора.
- `modify` — человек вводит новую правую часть; изменённое правило
  **заново проходит ту же верификацию** (semantic_check + is_size_safe),
  что и автоматические кандидаты — modify не обходит безопасность, а
  подаёт новый кандидат на проверку.

**`ual/control/psi.py` → `psi_step`** — диспетчер: `AUTO_ACCEPT` → `Ψ_agent`
(автокоммит), `NEEDS_REVIEW` → `Ψ_human` (реальный `input()` в терминале;
в демонстрациях подменяется скриптованными ответами для тестирования без
живого человека).

### Найденная и исправленная ошибка честности
В первой версии демо (`demo_action_pipeline.py`) в баннер для человека
попадала **захардкоженная заглушка `d=3.00`** вместо настоящего измеренного
значения — оператор увидел бы неверную цифру перед принятием решения.
Исправлено: `d` теперь считается ДО показа человеку через временный
коммит-и-откат (`env["__rules__"].append(rule)` → снимок → `pop()`),
и совпадает с итоговым значением при реальном коммите (`5.00` в обоих
местах после фикса).

**Результат демонстрации (5 NEEDS_REVIEW-кандидатов, все 4 Action_t
задействованы):** `accept` напрямую, `reject`, `request_explain`→`accept`
(с корректной обработкой опечатки на пути), `modify` с валидной заменой
(проходит semantic_check, но всё ещё требует финального подтверждения),
`modify` с неверной заменой (автоматически отклонено semantic_check без
дальнейших вопросов). Итог: 3 из 5 закоммичены, 2 отклонены.

## 20. `Goals` и `η` — от «можно ли менять» к «зачем менять»

Пользователь (и сторонний анализ, который он предоставил и попросил
критически разобрать) указал на пробел глубже, чем просто отсутствие
компонента `Goals`: пробел в **семантике существования системы**. Все
текущие величины (`semantic_check`, `is_size_safe`, `d`, `κ`, `Policy`,
`α`) отвечают на вопрос «можно ли изменить язык?». Ни одна не отвечает
на «зачем его менять?».

### Ключевой аргумент (проверен ниже кодом, не принят на слово)
Правила `(+ ?x 0) → ?x` и `(* ?x 1) → ?x` имеют одинаковые `d=3.5`,
одинаково безопасны и корректны — для текущей системы они неотличимы. Но
их полезность зависит от корпуса задач: если корпус состоит почти
целиком из сложений, первое правило полезнее; если из умножений —
второе. **Полезность — не свойство правила, а отношение `(rule, goal)`.**

### Уточнение терминологии (внесено при разборе, до кодирования)
Предложенная величина `η = ΔGoal / d` — **не физический КПД** (это
отношение величин в одних единицах, `[0,1]`), а **benefit-cost ratio**
(та же логика, что «QALY на доллар» в экономике здравоохранения) —
полезная эвристика для сравнения кандидатов между собой, не абсолютная
физическая величина.

### Связь с самого начала проекта
Выбор корпуса эталонных термов для `Goal.evaluator` — это ровно та же
проблема **grounding'а направленного интеллекта**, сформулированная в
разделе 1 (проблема №4, самое первое сообщение всего проекта). `Goal`
корректно локализует эту проблему в одном конкретном месте (выбор
корпуса), но не решает её — выбор корпуса всё ещё требует человека.

### Решение по архитектуре
`Goal` — объект (`name`, `evaluator(rules) -> score`, `weight`), а не
число, чтобы поддерживать несколько одновременных целей (многокритериальная
оптимизация вместо порогового if).

### Реализация (`ual/control/goals.py`) и найденная в процессе ошибка
Первая версия `Goal` считала **число шагов** переписывания до нормальной
формы («меньше шагов = лучше»). Проверка на реальных данных дала
**обратный результат**: без правил вообще любой терм тривиально уже
«нормален» за 0 шагов (ничего не может примениться) — рабочее правило
всегда добавляет шаги по сравнению с пустым набором, даже когда РЕЗУЛЬТАТ
становится проще. Метрика по числу шагов имела неверный знак у самой
своей конструкции. Исправлено: `Goal` теперь измеряет **размер результата**
после нормализации (`ast_size` нормальной формы), не путь до него —
переименовано в `MinimizeNormalFormSize`.

**Проверка утверждения на реальных данных (не на словах):** `d(add-zero)
= d(mul-one) = 3.50` (подтверждено), но на корпусе из сложений
`add-zero` даёт `ΔGoal=+4.20, η=+1.20`, а `mul-one` — `ΔGoal=0, η=0`; на
корпусе умножений — ровно наоборот. Утверждение «полезность — отношение
`(rule, goal)`, не свойство правила» подтверждено эмпирически.
